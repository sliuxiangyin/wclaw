# 插件开发文档（严格执行版）

本文是 `plugins/*` 插件的执行规范。目标是让任意插件都能在不依赖宿主内部实现的前提下，被宿主稳定加载和编排。

- 类型契约唯一来源：`@wclaw/plugin-sdk` 的 `src/runtime-contract.ts`
- 清单规范来源：`../../docs/项目功能/插件插件配置.md`
- 本文优先级：用于日常开发与评审的落地规范

---

## 1. 必须遵守的硬规则

以下规则不是建议，而是必须满足：

1. 入口必须是 ESM，且 `plugin.json.entry` 指向的模块必须 `export default class`。
2. 插件只能依赖 SDK 契约，禁止 `import` 宿主内部目录（`apps/host-api/**` 等）。
3. 禁止插件直连 MCP Server；必须走宿主注入能力（如 `invokeHostMcpTool`）。
4. `executeTurn` 若实现，返回值必须是 `PluginTurnHandleResult`，且 `text` 必填。
5. 宿主注入能力均视为“可选”，调用前必须判空或使用 guard。
6. 插件内不得假设宿主会对某个 `pluginId` 做特判。

任一项不满足，都视为“不合规插件”。

---

## 2. 标准目录与产物

推荐结构如下（最小可维护形态）：

```text
plugins/<plugin-id>/
├── plugin.json
├── package.json
├── src/
│   └── runtime.ts
├── dist/
│   └── runtime.mjs
└── README.md
```

要求：

- `plugin.json.entry` 必须和真实产物路径一致（例：`dist/runtime.mjs`）。
- 编译后产物必须可被 Node `import()` 正常加载。
- 若用 TypeScript，最终仍需输出 ESM（推荐 `.mjs` 产物或 `"type": "module"`）。

---

## 3. `plugin.json` 最小合规模板

以下字段必须与实际行为一致，禁止“先填上再说”：

```json
{
  "id": "demo-plugin",
  "displayName": "示例插件",
  "version": "0.1.0",
  "apiVersion": "v3",
  "kind": "runtime_plugin",
  "entry": "dist/runtime.mjs",
  "description": "示例插件描述",
  "triggerDescription": "进入 Chat 后发送 hello",
  "examples": [{ "input": "hello" }],
  "permissions": [],
  "sessionProvider": {
    "mode": "single"
  },
  "configSchema": {
    "type": "object",
    "properties": {}
  },
  "defaultConfig": {}
}
```

关键约束：

- `id`：全局唯一、kebab-case。
- `kind`：当前使用 `runtime_plugin` 或 `command_plugin`。
- `commandMode`：仅 `command_plugin` 必填，且仅允许 `ephemeral_no_context` / `ephemeral_with_context` / `isolated_chat`。
- `configSchema` 与 `defaultConfig` 必须可配套使用（默认值要满足 schema）。

## 4. 运行时类开发规范

插件入口类建议最小化状态、显式依赖：

```ts
import type {
  PluginRuntimeExtension,
  PluginRuntimeExtensionDeps,
  PluginTurnContext,
  PluginTurnHandleResult
} from "@wclaw/plugin-sdk";

export default class DemoRuntime implements PluginRuntimeExtension {
  constructor(private readonly deps: PluginRuntimeExtensionDeps) {}

  async executeTurn(ctx: PluginTurnContext): Promise<PluginTurnHandleResult> {
    ctx.emitAssistantDelta?.("开始处理…\n");
    return { text: `收到：${ctx.message}` };
  }
}
```

方法说明（均为可选）：

- `executeTurn`：单轮输入处理主入口。
- `executeCompleted`：宿主编排落库完成后的回流通知。
- `decorateSessions`：会话列表增强（展示属性）。
- `clearSession`：会话清理钩子。
- `getScheduledTasks` / `runScheduledTask`：周期任务声明与执行。

实现要求：

- 不要在 `executeTurn` 内阻塞长时间同步任务。
- 若返回 `continue: true`，表示允许宿主继续后续编排（如 LLM）。
- `persist` 中 `sessionId`、`role`、`content` 必须完整、可校验。

---

## 5. 可选注入能力的安全调用

`deps` 中这些能力必须按“可能不存在”处理：

- `ingestExternalUserTurn`
- `invokeHostMcpTool`
- `releaseHostMcpContext`
- `invokeHostLlm`

推荐：

- 使用 SDK guard（如 `guardInvokeHostMcpTool`、`guardInvokeHostLlm`）。
- 或显式判空：`if (typeof this.deps.invokeHostLlm === "function") { ... }`
- 能力不可用时返回可读文案，不要抛出难定位的底层错误。

---

## 6. 开发流程（必须按顺序）

1. 先写 `plugin.json`，明确 `kind` / `commandMode`（若为 `command_plugin`）与 permission。
2. 再写 runtime 类，最先打通 `executeTurn -> text` 最小链路。
3. 增加可选能力（LLM/MCP/调度）时，同步补充异常分支。
4. 本地构建，确保 `entry` 指向的产物真实存在且可加载。
5. 用宿主进行最小联调（至少验证一次 chat 回合）。
6. 完成下方“发布前验收清单”。

---

## 7. 发布前验收清单（DoD）

- [ ] `plugin.json` 字段与实际行为一致，且 `entry` 可加载。
- [ ] 入口模块是 `export default class`，可 `new` 成功。
- [ ] `executeTurn`（若实现）始终返回合法 `PluginTurnHandleResult` 且含 `text`。
- [ ] 所有宿主注入能力调用都做了 guard/判空。
- [ ] 使用了 MCP/LLM 时，失败路径能给出用户可理解提示。
- [ ] 调度任务（若实现）`intervalMs` 合理且执行逻辑可中断、可重试。
- [ ] 未引入宿主内部依赖、未出现插件特判耦合。
- [ ] 通过仓库约定检查（如 `pnpm lint:arch`）与最小联调。

---

## 8. 常见不合规写法（禁止）

- `export const runtime = ...`（错误：不是默认导出类）
- 在插件里 `import "../../../apps/host-api/src/..."`（错误：越界依赖）
- 直接请求 MCP 后端地址（错误：绕过宿主策略）
- `executeTurn` 返回字符串而非对象（错误：契约不匹配）
- 对 `invokeHostLlm!()` 非空断言直接调用（错误：未处理未注入场景）
