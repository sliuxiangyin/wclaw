# 设计模式与分层规范（防止代码堆叠）

目标：在开发早期就约束好结构，避免“所有逻辑写在一起”导致后期难维护。

---

## 1. 先定边界，再写功能

强制顺序：
1. 先确定模块边界（谁负责什么）。
2. 再定义接口契约（入参/出参/错误码）。
3. 最后实现业务逻辑。

禁止：
- 先在 `routes` 里写完全部业务，再回头“重构”。
- 前端页面里直接拼接网络请求、状态机、业务规则。

---

## 2. 后端推荐模式（Fastify）

分层职责固定：
- `routes`：只做路由绑定，不写业务。
- `controllers`：参数提取与返回封装。
- `services`：核心业务流程（可组合多个仓储）。
- `repositories`：只管数据存取（SQLite）。
- `core`：错误码、响应结构、基础能力。

推荐模式：
- **Repository Pattern**：隔离 SQLite 细节，避免 SQL 到处散落。
- **Service Layer Pattern**：业务流程放 service，route/controller 保持薄。
- **Strategy Pattern**：不同插件能力差异（single/multi session）用策略实现。
- **Factory Pattern**：插件运行时实例创建统一入口。
- **Adapter Pattern**：MCP/插件协议差异统一转换。

---

## 3. 前端推荐模式（React）

分层职责固定：
- `pages`：路由页面壳，不写复杂业务。
- `features/*/hooks`：状态与数据编排。
- `features/*/components`：业务组件渲染。
- `lib/api`：统一请求封装，不在页面里散落 fetch。
- `components/ui`：纯 UI 组件（shadcn）。

推荐模式：
- **Container / Presentational**：容器处理数据，展示组件只管渲染。
- **Hook as Service**：把业务逻辑封装到 hooks。
- **ViewModel Pattern（轻量）**：复杂页面先组装 view model 再渲染。

---

## 4. 插件体系模式（解耦重点）

核心规则：
- 宿主不写插件特判分支（如 `if pluginId === 'weixin-bridge'`）。
- 插件通过协议声明能力，宿主按协议驱动 UI 和流程。
- 多账号也只是“多会话”的一种实现。

推荐模式：
- **Plugin Driver Pattern**：插件提供 `sessionProvider`/capabilities，宿主只消费。
- **Command Pattern**：`/login`、`/command` 等都视为命令消息。
- **Event Pattern**：状态变化通过事件回传（登录成功、会话变更）。

### 4.1 宿主-插件通信总线（HPC）— 与插件代码的边界（写死）

全文以 `docs/项目功能/宿主插件通信总线/宿主-插件通信总线_设计文档.md` **§0、§1、§1.3** 为准。摘要：

- **不得**用 Hub 替代 **`PluginRuntimePort` 上对插件实例方法的直接调用** / 插件实例方法的**同步返回值**路径。
- **`emitPluginActivity` + AI Chat SSE** 与 **`ctx.publish`/Hub** 职责分离（§0.2）。

## 5. 反模式清单（必须避免）

1. 路由中直接写 SQL + 业务判断 + 响应拼装。
2. 前端页面组件里直接 `fetch` 多接口并处理所有状态。
3. 同一错误码在不同模块语义不一致。
4. 插件和宿主互相 import 内部代码。
5. 新需求直接改已有大文件，不拆模块。

---

## 6. 合并前检查清单（PR Checklist）

- [ ] 新增功能是否有明确模块归属？
- [ ] 是否新增/修改了契约文档？
- [ ] route/controller 是否保持“薄层”？
- [ ] SQLite 读写是否只在 repository 层？
- [ ] 前端业务逻辑是否在 feature hooks，而不是 page？
- [ ] 是否引入了插件特判分支？
- [ ] 错误响应是否统一 `ok/data/error/traceId`？

---

## 7. 渐进重构策略（当代码已经写乱）

1. 先冻结功能，不再继续堆在旧文件。
2. 抽出 repository（先把 SQL 拿出来）。
3. 抽出 service（再把业务流程拿出来）。
4. 最后精简 route/controller 与页面组件。
5. 每次只重构一个模块，避免大爆炸改动。
