# 宿主功能

| 功能模块 | 作用 | 当前状态 | 说明 |
|---|---|---|---|
| 插件功能（生命周期 + 会话 + 配置） | 负责插件发现、加载、能力声明校验、会话路由与配置管理。 | 已落地（核心） | 宿主启动时扫描 `plugins/*/plugin.json`，校验 `PLUGIN_SPEC_V3`；提供插件列表、详情、配置读写与校验接口；按插件声明的 `sessionProvider` 与 `capabilities` 驱动会话与交互，避免插件特判。 |
| MCP 接入（Host Gateway） | 统一提供 MCP 目录探测、工具 schema 查询、参数校验与代调用入口。 | 设计已明确，接口为 P2 优先级 | 插件不得直连 MCP，只能调宿主接口（如 `GET /api/mcp/catalog`、`POST /api/mcp/invoke`）；调用链要求可审计，结合策略层做权限过滤与风险控制。 |
| Scheduler 调度器 | 统一承载运行时插件的轮询/定时任务，控制并发、重试、退避，降低内存与 GC 风险。 | 已落地（v1） | 启动时注册插件声明的 `getScheduledTasks`；周期 tick 内 `runScheduledTask`（如微信桥 `poll-inbox`）；编排租约 `POST /api/orchestration/lease/grant|revoke` 与 `GET /api/orchestration/scheduler/status` 已接线。熔断、全局限流与可观测仍为后续迭代。 |
| LLM 配置中心 | 统一管理宿主的 LLM Provider、模型与自定义配置。 | 已落地（P0） | 提供 `GET /api/llm/config`、`PUT /api/llm/config`；支持自定义模型发现接口（`POST /api/llm/providers/custom/models:discover`，可选）；前端在设置页统一配置。 |
| Notification SSE 长连接通知 | **Host Event Hub 的一个下游 Sink**：向前端推送「允许上控制台」的事件。 | 已落地（Phase 1） | `GET /api/notifications/stream`；scope 过滤、心跳、断开清理。与聊天回合 SSE 解耦；**不是**插件→宿主的总线本体。 |
| 宿主-插件通信总线（HPC Bus） | **Ingress API**（插件上报抽象）+ **Host Event Hub**（唯一入站、扇出）+ 多 Sink（含 Notification SSE）。 | 设计已确认 / 实现渐进 | 当前 v1 仍有生产者直连 `notification-bus` 的过渡路径；目标为插件只走 Ingress，Hub 再分发给 SSE 与其它内部消费者。 |

## 1) 插件功能（宿主视角）

| 子能力 | 对应作用 | 关键约束 |
|---|---|---|
| 插件发现与加载 | 扫描插件目录并动态加载运行时入口。 | `entry` 必须位于插件目录内，禁止 `../` 路径逃逸。 |
| 清单校验 | 保证插件声明规范一致。 | 先 JSON Schema，再语义校验；未通过即拒载但不影响其他插件。 |
| 能力驱动路由 | 按 `capabilities` 判断 chat/command/mcp 等能力。 | 禁止宿主出现 `pluginId === xxx` 特判分支。 |
| 会话管理 | 支持 single/multi 会话模式，统一会话协议。 | `chat=true` 时必须声明 `sessionProvider`。 |
| 配置中心 | 通过 `configSchema` 渲染并校验插件配置。 | 流程为先 `validate`，后 `PUT` 保存。 |

### command_plugin 与 runtime_plugin（宿主如何理解）

- `command_plugin` 的 `ephemeral_no_context` / `ephemeral_with_context` / `isolated_chat` 属于命令执行模式：只有命中该插件命令时才执行。
- `ephemeral_no_context` 不接入 LLM；`ephemeral_with_context` 与 `isolated_chat` 可接入 LLM，区别在于是否使用隔离会话上下文。
- `sessionProvider.mode=single/multi` 表达会话数量形态，不等同于 `command_plugin` 的执行模式。
- `runtime_plugin` 常见为 `multi` 会话（如微信桥按账号拆分多个会话），并在策略允许下可编排其他插件。
- 编排能力建议以 `capabilities.crossPluginInvoke` 与 `capabilities.orchestration` 为宿主判定依据。

## 2) 提供接入 MCP（宿主对插件）

| 接口能力 | 作用 | 备注 |
|---|---|---|
| `GET /api/mcp/catalog` | 获取当前可见 MCP servers/tools 目录。 | 应经过策略过滤后返回。 |
| `GET /api/mcp/tools/:toolId/schema` | 获取工具 schema，供插件参数表单渲染。 | 统一规范化 schema。 |
| `POST /api/mcp/validate` | 调用前参数校验（schema + policy）。 | 提前失败，减少无效调用。 |
| `POST /api/mcp/invoke` | 宿主代插件调用 MCP 工具。 | 记录 `traceId/sessionId/pluginId` 便于审计。 |

> 硬规则：插件禁止直连 MCP Server、禁止自行管理 MCP 凭据。

## 3) Scheduler 调度器（runtime_plugin 稳定性）

| 能力点 | 目的 | 建议基线 |
|---|---|---|
| 统一任务调度 | 替代插件内部无限循环。 | 用调度器接管轮询与定时任务。 |
| 并发与队列控制 | 防止任务风暴与资源打满。 | 设置 `concurrency` 上限与排队策略。 |
| 重试退避机制 | 降低瞬时失败导致的抖动。 | 配置 `maxRetry + backoff`。 |
| 超时与熔断 | 防止慢任务拖垮全局。 | 任务级 `timeout` + 插件级熔断降级。 |
| 资源治理 | 避免内存持续增长。 | 缓存要求 `TTL + maxSize`，并支持健康巡检。 |

## 4) LLM 配置（宿主统一入口）

| 能力点 | 作用 | 当前文档基线 |
|---|---|---|
| LLM 配置读取 | 让前端设置页拉取当前配置。 | `GET /api/llm/config` |
| LLM 配置保存 | 更新宿主使用的 provider/model/参数。 | `PUT /api/llm/config` |
| 自定义模型探测（可选） | 根据自定义 provider 拉取模型列表。 | `POST /api/llm/providers/custom/models:discover` |
| 与插件关系 | 插件声明是否具备 llm 能力。 | `capabilities.llm` 控制插件可用能力面。 |

## 5) 宿主能力边界（便于对齐）

- 宿主负责：插件生命周期、会话与路由、策略鉴权、MCP 代理、调度治理、配置中心与审计日志。
- 插件负责：按声明实现业务逻辑，通过宿主公开 API 获取能力，不依赖宿主内部实现。
- 前端负责：按宿主协议渲染插件与配置，不写插件特判 UI 流程。

## 6) Notification SSE（Hub 的 Sink → 前端）

| 能力点 | 作用 | 当前状态 |
|---|---|---|
| 常驻事件通道 | 前端建立一次连接持续收通知。 | `GET /api/notifications/stream` 已可用 |
| 过滤订阅 | 按 `scope=global/plugin/session` 接收事件。 | 已实现基础过滤 |
| 心跳保活 | 防止长连接被代理空闲断开。 | 已实现 `: ping` 心跳 |
| 事件来源 | 来自 **Host Event Hub 允许透出** 的子集（过渡：部分仍由 `notification-bus` 直连）。 | scheduler 等已接入 |

> 与 `POST /api/ai/chat` 的单回合 SSE 区分：本通道是 **控制台通知 Sink 的 HTTP 写端**；聊天流是请求生命周期内的响应流。  
> **背压**：无对外开放事件 API，仍须防范进程内高频与慢客户端导致的写端积压（见 HPC 设计文档 §6）。

## 7) 宿主-插件通信总线（HPC Bus）

**写死规则全文**：`docs/项目功能/宿主插件通信总线/宿主-插件通信总线_设计文档.md` **§0、§1、§1.3**。

| 能力点 | 作用 | 约束 |
|---|---|---|
| **`hostEventHub.publish`** | 通信中心入站；**`HOST_EVENT_TOPICS`**；支持 **多 topic**。 | 插件禁止直连；插件经 **`ctx.publish`** |
| **`publishToNotificationStream`**（薄封装） | 固定 **Notification** topic，便于调度器等。 | 内部仍走 Hub |
| **`ctx.publish`** | `PluginChatContext` / `PluginScheduledTaskContext` 上**可选**注入（§0.1.1）。 | 唯一允许的 ctx 扩展名（HPC） |
| **Host Event Hub** | 按 topic 扇出到 Sink（含 **`publishNotification`** 链）。 | 不经 Hub：同步 **return** |
| **`emitPluginActivity`** | 仅 **AI Chat SSE**。 | 与 Hub **`publish`** 并存、职责分离 |

## 8) 术语对照（宿主视角）

| 概念 | 宿主如何使用 | 说明 |
|---|---|---|
| `command_plugin` 执行模式 | 决定命令触发后的执行路径与上下文策略。 | `ephemeral_no_context` / `ephemeral_with_context` / `isolated_chat`。 |
| `sessionProvider.mode` | 决定宿主展示和路由会话列表的方式。 | `single`（单会话）或 `multi`（多会话，如微信桥按账号）。 |
| `capabilities.llm` | 决定宿主是否允许该插件走 LLM 调用路径。 | 由插件声明并受策略控制。 |
| `capabilities.crossPluginInvoke` | 决定宿主是否允许该插件编排调用其他插件。 | `none` / `declared` / `policy_granted`。 |
| `capabilities.orchestration` | 决定宿主是否将其纳入编排与租约治理流程。 | 常用于 `runtime_plugin`。 |
