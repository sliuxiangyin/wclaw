# 宿主功能总览（Host）

本文用于统一说明宿主（Host）当前承担的核心能力，重点覆盖：

- 插件管理（Plugin）
- 调度与编排（Scheduler / Orchestration）
- LLM 配置（LLM Config）
- Chat 会话与消息流（Chat）

状态口径：

- `✅ 已实现`：代码中已有可用能力（含最小实现）。
- `🔳 规划中`：文档定义过，但当前版本未完整落地。

---

## 1. 插件管理（Plugin）

### 1.1 功能目标

宿主负责插件全生命周期管理：发现、校验、加载、配置、能力暴露与会话接入。插件通过 `plugin.json` 声明能力，不允许与宿主内部强耦合。

### 1.2 当前能力

- ✅ 插件目录扫描与基础加载（基于 `plugins/*/plugin.json`）
- ✅ 插件列表与详情查询
- ✅ 插件配置读写
- ✅ 插件配置校验接口（当前为最小占位实现）
- ✅ 插件规范校验接口（Plugin Spec Validate）
- ✅ 插件 Chat / Command 入口对接

### 1.3 主要接口（已存在）

- `GET /api/plugins`
- `GET /api/plugins/:pluginId`
- `GET /api/plugins/:pluginId/config`
- `PUT /api/plugins/:pluginId/config`
- `POST /api/plugins/:pluginId/config/validate`
- `POST /api/plugins/:pluginId/validate`
- `GET /api/plugins/:pluginId/sessions`
- `GET /api/plugins/:pluginId/sessions/:sessionId/messages`（合并 `plugin_chat_messages` + `plugin_chat_activity` 时间线，`?limit=` 默认 100、最大 500）
- `POST /api/plugins/:pluginId/sessions/:sessionId/switch`
- `POST /api/plugins/:pluginId/chat`
- `POST /api/plugins/:pluginId/command`

### 1.4 已知差距

- 🔳 插件配置校验接口仍为占位逻辑（需升级为 schema + 语义双校验）。

---

## 2. 调度与编排（Scheduler / Orchestration）

### 2.1 功能目标

宿主统一承担跨插件编排、会话流转、任务调度与治理，避免插件内部裸轮询或无界并发，保证可控与可观测。

### 2.2 当前能力

- ✅ Chat 侧已具备基础编排能力（宿主路由统一接入并分发至对应执行路径）
- ✅ 文档层已明确编排分层：Orchestration / Execution / Session / Persistence
- ✅ 编排租约：`POST /api/orchestration/lease/grant`、`POST /api/orchestration/lease/revoke`（供调度 tick 等场景使用）
- ✅ 宿主调度器 v1：启动/关闭时 runner 生命周期；从插件加载 `getScheduledTasks`，按 `intervalMs` 触发 `runScheduledTask`；`GET /api/orchestration/scheduler/status` 可观测 tick 与任务概况
- ✅ 微信桥等 runtime 已通过调度任务替代插件内无限 `while` 常驻拉取（单次 tick 内拉取）

### 2.3 规划中的平台能力

- 🔳 调度治理增强：并发上限、任务级退避/重试默认值、熔断与全局限流统一策略
- 🔳 调度与租约的可观测性（指标、结构化日志、trace 串联）

### 2.4 治理原则（必须坚持）

- 插件不自行实现无限轮询；调度权归宿主。
- 编排与执行解耦，不做插件 ID 特判分支。
- 调用链需可追踪（`traceId/sessionId/pluginId`）。

## 3. LLM 配置（LLM Config）

### 3.1 功能目标

宿主统一管理模型供应商、模型名、参数与默认策略，前端通过管理台配置，Chat 执行时统一读取，避免各插件各自维护配置。

### 3.2 当前能力

- ✅ LLM 配置查询
- ✅ LLM 配置更新
- ✅ 管理台页面可编辑 LLM 基础配置

### 3.3 主要接口（已存在）

- `GET /api/llm/config`
- `PUT /api/llm/config`

### 3.4 后续增强方向

- 🔳 多模型路由策略（按插件/场景选择模型）
- 🔳 更细粒度参数策略（token 限额、温度区间、超时分级）
- 🔳 配置变更审计与回滚

---

## 4. Chat（会话与消息流）

### 4.1 功能目标

宿主统一承接 Chat 请求，完成消息编排、插件执行、上下文并入、事件沉淀与前端展示，形成“runtime_plugin 主会话 + command_plugin 分流执行”的体系。

### 4.2 当前能力

- ✅ `POST /api/ai/chat`：统一 AI Chat 入口
- ✅ 支持普通 JSON 响应
- ✅ 支持 `text/event-stream`（SSE）分块响应
- ✅ `GET /api/ai/events`：事件查询（基础能力）
- ✅ 插件 Chat 入口可与宿主编排对接
- ✅ 消息流程与模式文档已形成（runtime_plugin / command_plugin 多模式）
- ✅ **外部进线**：向 `runtime_plugin` 注入 **`ingestExternalUserTurn`**，与 **`POST /api/ai/chat`** 同源编排；成功后可选 **`reflowChatToChannel`**；经 **`hpc.chat`** 推送 **`chat.session.updated`**。清单与口径：`docs/进度/外部进线-ingest检查清单.md`；微信桥字段：`docs/weixin_bridge_api_contract_微信桥接口契约.md`。

### 4.3 主要接口（已存在）

- `POST /api/ai/chat`
- `GET /api/ai/events`
- `GET /api/plugins/:pluginId/sessions/:sessionId/messages`
- `POST /api/plugins/:pluginId/chat`
- `POST /api/plugins/:pluginId/command`

### 4.4 后续增强方向

- 🔳 SSE 事件规范进一步收敛（事件命名与载荷标准化）
- 🔳 command_plugin 三模式在生产场景下的稳定性增强
- 🔳 消息持久化与审计链路的完整闭环（含异常补偿）

## 5. 宿主能力边界（统一口径）

- 宿主负责：插件生命周期、权限策略、编排调度、LLM 配置、MCP 代理入口、审计与治理。
- 插件负责：按清单声明能力、实现业务逻辑、通过宿主公开 API 使用平台能力。
- 明确禁止：插件直连 MCP、插件依赖宿主内部模块、宿主写插件 ID 特判逻辑。

---

## 6. 现阶段结论（给产品/开发同步）

当前宿主已经具备“可运行的基础闭环”：

1. 插件可被发现、配置、接入 chat；
2. Chat 与 LLM 配置链路可跑通；
3. 管理台可完成基础运维动作。

下一阶段优先项：

1. 补齐编排租约与调度治理能力；
2. 完成 `workspace-echo` 运行时落地；
3. 提升 Chat 事件规范与可观测性，形成生产级稳定基线。

## 1) 插件功能（宿主视角）

| 子能力 | 对应作用 | 关键约束 |
|---|---|---|
| 插件发现与加载 | 扫描插件目录并动态加载运行时入口。 | `entry` 必须位于插件目录内，禁止 `../` 路径逃逸。 |
| 清单校验 | 保证插件声明规范一致。 | 先 JSON Schema，再语义校验；未通过即拒载但不影响其他插件。 |
| 能力驱动路由 | 按 `capabilities` 判断 chat/command/mcp 等能力。 | 禁止宿主出现 `pluginId === xxx` 特判分支。 |
| 会话管理 | 支持 single/multi 会话模式，统一会话协议。 | `chat=true` 时必须声明 `sessionProvider`。 |
| 配置中心 | 通过 `configSchema` 渲染并校验插件配置。 | 流程为先 `validate`，后 `PUT` 保存。 |

## 2) 提供接入 MCP（宿主对插件）

| 接口能力 | 作用 | 备注 |
|---|---|---|
| `GET /api/mcp/catalog` | 获取当前可见 MCP servers/tools 目录。 | 应经过策略过滤后返回。 |
| `GET /api/mcp/tools/:toolId/schema` | 获取工具 schema，供插件参数表单渲染。 | 统一规范化 schema。 |
| `POST /api/mcp/validate` | 调用前参数校验（schema + policy）。 | 提前失败，减少无效调用。 |
| `POST /api/mcp/invoke` | 宿主代插件调用 MCP 工具。 | 记录 `traceId/sessionId/pluginId` 便于审计。 |

> 硬规则：插件禁止直连 MCP Server、禁止自行管理 MCP 凭据。

## 3) Scheduler 调度器（runtime_plugin 稳定性）

| 能力点 | 目的 | 建议基线 |
|---|---|---|
| 统一任务调度 | 替代插件内部无限循环。 | 用调度器接管轮询与定时任务。 |
| 并发与队列控制 | 防止任务风暴与资源打满。 | 设置 `concurrency` 上限与排队策略。 |
| 重试退避机制 | 降低瞬时失败导致的抖动。 | 配置 `maxRetry + backoff`。 |
| 超时与熔断 | 防止慢任务拖垮全局。 | 任务级 `timeout` + 插件级熔断降级。 |
| 资源治理 | 避免内存持续增长。 | 缓存要求 `TTL + maxSize`，并支持健康巡检。 |

## 4) LLM 配置（宿主统一入口）

| 能力点 | 作用 | 当前文档基线 |
|---|---|---|
| LLM 配置读取 | 让前端设置页拉取当前配置。 | `GET /api/llm/config` |
| LLM 配置保存 | 更新宿主使用的 provider/model/参数。 | `PUT /api/llm/config` |
| 自定义模型探测（可选） | 根据自定义 provider 拉取模型列表。 | `POST /api/llm/providers/custom/models:discover` |
| 与插件关系 | 插件声明是否具备 llm 能力。 | `capabilities.llm` 控制插件可用能力面。 |

## 5) 宿主能力边界（便于对齐）

- 宿主负责：插件生命周期、会话与路由、策略鉴权、MCP 代理、调度治理、配置中心与审计日志。
- 插件负责：按声明实现业务逻辑，通过宿主公开 API 获取能力，不依赖宿主内部实现。
- 前端负责：按宿主协议渲染插件与配置，不写插件特判 UI 流程。
