# 宿主-插件通信总线（HPC Bus）设计文档

## 0. 硬性规则（写死，禁止违反）

以下条款用于防止后期实现「绕 Hub 过度耦合」或「借 Hub 改掉稳定调用链」。**若代码与本文冲突，以本文为准**；改代码前须先评估并更新相关文档。

### 0.1 宿主 → 插件（`loadPluginRuntimeExtension`）

- 宿主通过 **`loadPluginRuntimeExtension`** 动态 `import` 插件入口后，对 **`export default` 类** 执行 **`new`**，再对实例上的 **`handleChat`、`runScheduledTask`、`executeCommand`、`decorateSessions`、`clearSession`、`getScheduledTasks`** 等调用，**必须**仍是**同步栈上的直接方法调用**：**不得**经 Host Event Hub 绕一圈再回调用方；**返回值 / 异常**仍在原调用栈，**不经 Hub**。
- **上下文（ctx）契约**：除下文 **§0.1.1 唯一例外**外，**禁止**为其它目的增删改既有字段；若需更多字段须 ADR。

#### 0.1.1 唯一允许的 ctx 扩展：`publish`（已批准）

- 允许在 **`PluginChatContext`、`PluginScheduledTaskContext`**（以及后续若扩展同类「宿主调用插件」ctx，须同步列于此）上**仅增加可选方法**：`publish?: HostHubPublishFn`。
- **`publish` 语义**：宿主注入的闭包，内部调用 **`host-event-hub.service.ts` 的 `publish`**；插件**禁止**自行实现、禁止 import Hub。
- **不得**在同一 ctx 上再堆叠其它 HPC 专有 API（如第二套 `emitX`），除非单独 ADR。

### 0.2 `emitPluginActivity` 与 AI Chat SSE

- **`emitPluginActivity` + `POST /api/ai/chat` 流式 SSE**（含 `plugin-activity` chunk、相关持久化）为**独立通道**。
- **禁止**为「对齐 HPC」而无评审地改写该链；**不要求**把该流复制进 **`GET /api/notifications/stream`**（常驻 Notification SSE）。

### 0.3 Host Event Hub 的定位（主要收什么）

- Hub 承接：**宿主自身**产生的、需多 Sink 扇出的可观测事件；以及**在插件同步 `handleChat`/`runScheduledTask` 调用栈之外**产生、且需进宿主统一分发的事件（例如桥接层异步回调、宿主侧 adapter 汇总后再上报）。
- **不得**把 Hub 设计成「替代 `loadPluginRuntimeExtension` / 插件实例方法返回值」或「替代 AI Chat SSE 内 `emitPluginActivity`」的默认路径。

### 0.4 插件 → 宿主（进 Hub 的允许方式）

- 插件 **禁止** `import` 宿主内部实现、**禁止**直连 `notification-bus` 或 Notification 路由。
- 允许方式（可并存）：
  1. **`ctx.publish?.(...)`**（在 §0.1.1 涉及的调用路径上，由宿主注入）；
  2. **宿主胶水层**（`apps/host-api` 内 adapter）在栈外回调中调用 **`hostEventHub.publish`**；
  3. **宿主业务模块**直接调用 **`hostEventHub.publish`** 或 **`notification` 模块对外暴露的 `publish`**（见 §1.2）。

### 0.5 Notification SSE

- **`GET /api/notifications/stream`** 仍为 Hub 的 **Sink 之一**；实现可保留 `notification-bus.service.ts` 作为该 Sink 背后的内存分发，**不等于** Hub 本体。

### 0.6 架构脚本与插件特判

- 仍须遵守 `pnpm lint:arch`：**禁止**宿主 `pluginId === '…'` 特判；HPC 相关路由/topic 亦不得成为特判借口。

## 1. 通信中心需求整理（产品口径）

下列条目与实现文件对应关系在 §1.2；**注入方案**见 §1.3。

| # | 需求 | 说明 |
|---|------|------|
| 1 | **`host-event-hub.service.ts`** | 宿主侧通信中心：统一 **`publish` 入站**、按 **topic** 扇出到已注册处理逻辑（Sink / 内部订阅者）。 |
| 2 | **暴露 `publish`** | 宿主代码与胶水层统一通过 **`hostEventHub.publish(...)`**（命名以代码为准）入站；**禁止**业务散落直连旧 `notification-bus`（迁移期除外）。 |
| 3 | **Notification 通道也暴露 `publish`** | 在 `services/notification/` 增加薄封装（如 **`publishToNotificationStream`**），内部固定选用 **`HOST_EVENT_TOPICS.Notification`** 调 Hub，便于调度器等只关心「上控制台」语义而不重复写 topic。 |
| 4 | **向插件注入 Hub 的 `publish`** | 见 **§1.3**；落点：`plugin-chat.service`、`scheduler-runner`（及今后同类调用点）构造 ctx 时注入。 |
| 5 | **topic 常量** | 在 **`host-event-hub.topics.ts`**（或与 hub 同目录）定义 **`HOST_EVENT_TOPICS`**，例如 `Notification`、`Chat`（值为稳定字符串，用于分发与治理）。 |
| 6 | **多 topic 一次调用** | `publish` 支持 **`topics: (keyof typeof HOST_EVENT_TOPICS)[]`**（或等价只读元组）；Hub **按 topic 逐个分发**到对应订阅（同一 payload 可送多处；各 Sink 自行决定是否处理）。 |

**与 `emitPluginActivity` 的关系**：**不替代** §0.2；`publish` 走 Hub 与常驻 Notification / 未来 Chat 等 Sink；`emitPluginActivity` 仍只服务 AI Chat 请求内 SSE。

## 1.1 总体结构（已确认）

采用**三层分离**，避免把「Notification SSE」与「插件→宿主通信」混为同一概念：

```
插件（及宿主其它生产者）
        │
        ▼ 仅使用宿主提供的抽象上报入口（Ingress API）
┌───────────────────────────────────────┐
│  宿主事件接收中心（Host Event Hub）    │  ← 唯一入站枢纽：校验、归一化、路由、扇出
│  （进程内实现可演进为队列/外部总线）    │
└───────────────────────────────────────┘
        │ 分发给多个宿主内部消费者（Sinks），彼此解耦
        ├──► Notification SSE 适配器 → 浏览器（仅「需上控制台」的事件）
        ├──► 观测 / 指标（后续）
        ├──► 审计 / 持久化（按需）
        └──► 编排与其它宿主子系统（按需）
```

> 上图仅描述 **事件上报（Ingress）** 路径。**宿主经 `loadPluginRuntimeExtension` 调用插件实例**（`handleChat` 等）不在此图中、**不经 Hub**。

**硬性边界（与 §0 一致，摘要）**

- **插件 → 宿主（进 Hub）**：插件不直连 SSE / `notification-bus`；可用 **`ctx.publish`**（§0.1.1）或胶水层 / 宿主直接调 Hub（§0.4）。
- **Host Event Hub**：宿主侧入站事件的**语义与路由**层；**不**替代 §0.1 返回值路径、§0.2 `emitPluginActivity` 链。
- **Notification SSE**：仅为 Hub 的**一个下游 Sink**；**不是**总线本体。

**Hub 不替代「请求—响应」主路径（避免过度设计）**

- **经 `loadPluginRuntimeExtension` 的宿主 → 插件调用**（`handleChat`、`runScheduledTask`、`executeCommand`、`decorateSessions`、`clearSession`、`getScheduledTasks` 等）**全程不走 Host Event Hub**：仍是动态 `import` 后 **`new` 默认导出类** 再对实例方法的**直接调用**；入参、返回值、异常均在原调用栈上处理。
- **Host Event Hub 仅服务于「事件上报」入站**（Ingress）：与上述导出函数调用链**正交**；不要求、也不应把「加载模块 / 调插件」绕经 Hub。
- Hub 承接**与主返回值正交的「事件」**（§0.3）；**不包括** AI Chat SSE 路径上的 `emitPluginActivity`（§0.2）。**业务结果**仍由 `handleChat` 等 **return** 表达。

调度器等已向 **`hostEventHub.publish` / `publishToNotificationStream`** 收敛；内向 **`dispatchNotificationToSubscribers`** 仅 Hub Sink 调用（见 §1.2.1）。改造清单见《宿主-插件通信总线_改造文档.md》。

### 1.2 模块与 API（实现对应）

| 路径（建议） | 职责 |
|--------------|------|
| `services/host-event-hub/host-event-hub.service.ts` | 通信中心：**`publish`**、topic 订阅注册、**多 topic** 时按 topic **逐个分发**。 |
| `services/host-event-hub/host-event-hub.topics.ts` | **`HOST_EVENT_TOPICS`** 常量（如 **`Notification`**、**`Chat`**，值为稳定字符串）。 |
| `services/notification/*.ts`（薄封装） | 对外 **`publishToNotificationStream`** 等；内向实现仍在 **`notification-bus`**。 |
| `providers/host-event-hub-provider/*` | **Hub Provider（组合根）**：各域 **`registerSink`** 接线；入口 **`registerAllHostEventHubProviders()`**（见 **`host-event-hub_providers_设计.md`**）。 |
| `services/notification/notification-bus.service.ts` | 内向实现 **`dispatchNotificationToSubscribers`**；由 **`register-notification-hub-sink`** 注册到 Hub，**不**作业务直调入口。 |

#### 1.2.1 常驻 Notification SSE 投递唯一入口（写死）

- **对外（推荐）**：宿主业务模块使用 **`hostEventHub.publish`**（多 topic）或 **`publishToNotificationStream`**（仅 Notification 语义糖）。
- **内向**：**`dispatchNotificationToSubscribers`** 在 `notification-bus.service.ts`；**注册方式**：**`notificationHubProvider`** 在 **`attach(registrar)`** 内 **`registrar.registerSink(HOST_EVENT_TOPICS.Notification, …)`**；应用 **`hostEventHub.registerProvider(notificationHubProvider)`** 完成注入（**Hub 不 import** `notification-bus` 运行时）。详见 **`host-event-hub_providers_设计.md`**。
- **约定**：业务不直接 `import` **`dispatchNotificationToSubscribers`**；该符号仅出现在 **`notification-bus.service.ts`** 与 **`register-notification-hub-sink.service.ts`**（Hub Sink 闭包）。禁止再以 **`publishNotification`** 作为业务对外入口名。

**多 topic 语义**：一次 `publish({ topics: [A, B], notification?: ... })` → Hub 对 **A、B** 各触发一轮 Sink；含 **Notification** 时须带 **`notification`** 字段（与现有 `NotificationEvent` 形状兼容）。

### 1.3 向插件注入 Hub 的 `publish`（解决方案）

**目标**：插件侧只调用 **`ctx.publish(...)`**，不 import 宿主模块。

**做法（闭包注入）**：

1. **类型**：在 `plugin-runtime.loader.ts` 的 **`PluginChatContext`、`PluginScheduledTaskContext`** 上增加可选 **`publish?`**，类型为宿主定义的 **`HostHubPublishFn`**（参数含 **`topics`**、`level`、`scope?`、`payload?`；与 §1 表格一致）。
2. **谁构造**：**仅宿主**在调用插件前创建闭包：
   - **`plugin-chat.service.ts`**：`sendPluginChat` 内组装 ctx 时设置  
     `publish: (input) => hostEventHub.publish({ ...input, source: { pluginId, module: 'plugin-runtime' } })`，并按需写入 **`scope.sessionId`**。
   - **`scheduler-runner.service.ts`**：调用 **`runScheduledTask`** 前对当前 **`pluginId`** 注入同一形态的 **`publish`**（无 session 则可省略 `scope.sessionId`）。
3. **填写规则**：`pluginId`（及可选 `sessionId`）**由宿主注入**，插件**不得**伪造其它插件的 `pluginId`（宿主闭包已绑定）。
4. **可选扩展**：若 **`executeCommand`** 也需要内向事件，再在 **`PluginCommandContext`** 上增加 **`publish?`**（须单独列入清单）。

**非方案**：插件 `import` Hub；插件直接调 `notification-bus`；在 `plugins/*` 内读取宿主 env 拼 URL。

## 2. 事件模型（Envelope）

Hub 内统一使用稳定 envelope（与对外 SSE 的 `data` JSON 可对齐字段，但 Hub 先于 SSE 存在）：

```ts
type HpcEvent = {
  id: string;
  ts: string;
  source: {
    pluginId?: string;
    module: "scheduler" | "ai-chat" | "plugin-runtime" | "system" | string;
  };
  topic: string;
  level: "debug" | "info" | "warn" | "error";
  scope?: {
    pluginId?: string;
    sessionId?: string;
    accountId?: string;
  };
  payload?: Record<string, unknown>;
};
```

约束：

- `topic` 为稳定契约；需上 UI 的 topic 由 **SSE Sink** 侧策略决定是否转发、是否节流。
- `payload` 可扩展，禁止包含敏感明文（token、凭据、`context_token` 等）。

## 3. 订阅与扇出模型（宿主内部）

- Hub 维护**注册表**：各 Sink 声明感兴趣的 `topic` 前缀、`scope` 或策略函数。
- **SSE Sink** 仅订阅「允许透出到控制台」的事件子集；其它 Sink 可订阅全量或另一子集。
- 与浏览器相关的过滤（`global` / `plugin` / `session`）属于 **SSE 连接层**，不改变 Hub 作为单一入站的事实。

匹配原则（SSE 连接层，与现实现一致即可）：

- `scope=global`：该连接可接收允许透出的事件中、与连接 filter 匹配的部分。
- `scope=plugin`：匹配 `event.scope.pluginId`。
- `scope=session`：同时匹配 `pluginId + sessionId`。

## 4. 与 Notification SSE 的关系

| 组件 | 职责 |
|------|------|
| **Host Event Hub** | 接收、校验、扇出；**不**承担 HTTP/SSE 协议细节。 |
| **Notification SSE** | Hub 的一个消费者；将已选事件按 SSE 规范写入长连接。 |

详见《notification_sse_长连接通知通道设计.md》（仅描述浏览器通道与连接治理）。

## 5. 与插件运行时的关系

- **同步调用链**：见 **§0.1**；返回值不经 Hub。
- **`emitPluginActivity`**：仅 **AI Chat SSE**（§0.2）；与 **`ctx.publish`** 并存时职责分离：**前者**服务单次聊天响应流，**后者**服务 Hub 扇出（常驻通知、Chat topic 等）。
- **插件内向 Hub**：优先 **`ctx.publish`**（§0.1.1、§1.3）；栈外且**无 ctx** 的路径仍由 **宿主胶水层** 调 **`hostEventHub.publish`**。

## 6. 背压与频率（已确认）

**结论**：项目**不对外提供**「任意客户端可调的事件 HTTP API」，主要降低**被公网滥用**的风险；**不能**据此认为不存在性能与背压问题。

仍需关注：

1. **进程内生产者**：宿主模块与胶水层向 Hub 发布时仍可能高频；调度器、其它循环逻辑同样可能高频产出（**不**把「经 §0.2 保留的 `emitPluginActivity`」算入 Hub 背压主因，除非将来显式双写）。
2. **Hub 自身**：扇出应保持轻量；避免在 Hub 同步路径上对每条事件执行重 I/O（大段 SQLite、同步写盘）。
3. **SSE Sink**：浏览器读慢时，写缓冲区可能积压——**背压主要出现在 SSE 这一 Sink**，与是否存在开放 API 无关；需保留丢弃、断连或节流策略。
4. **工程策略**：UI 类事件可合并/采样；原始日志不进 SSE；重活异步化或独立队列。

## 7. 生命周期与可靠性

### 单进程实现（当前）

- Hub 可用内存结构 + 订阅者列表；无持久化、无补偿重放。
- 适合开发与单实例部署。

### 多实例演进（后续）

- Hub 后端可迁移到 Redis/NATS 等。
- SSE 侧可叠加短期缓存 + `Last-Event-ID` 补偿。
- 统一观测指标（连接数、丢弃数、延迟）。

---

## 8. 安全与治理

- payload 脱敏；topic 白名单（插件可发布范围受策略约束，后续）。
- Hub 入站校验：`pluginId`、payload 大小、类型与权限。

---

## 9. 设计决策（关键）

- **入站单一**：对外统一 **`hostEventHub.publish`**；Notification 语义可通过 **`publishToNotificationStream`** 薄封装；插件经 **`ctx.publish`**（§0.1.1）。
- **topic 常量**：`HOST_EVENT_TOPICS` 集中定义，多 topic 一次调用、Hub 内逐 topic 分发（§1）。
- **SSE 非核心**：Notification SSE 仅为订阅 **`HOST_EVENT_TOPICS.Notification`** 的 Sink 链终点。
- **渐进迁移**：调度器等宿主模块迁 Hub；**不**改写 **`emitPluginActivity` + AI Chat SSE**（§0.2）。
