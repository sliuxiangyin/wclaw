# 目录架构（前后端）

目标：给出可直接开工的目录结构，避免前后端混在一起。

---

## 1. 根目录

```txt
wclaw-weixing-v3/
  apps/            # 前后端应用
  packages/        # 公共能力包（含 @wclaw/plugin-sdk：插件 runtime 类型契约）
  plugins/         # 插件实现
  docs/            # 文档
  scripts/         # 脚本
  var/             # 运行数据（sqlite/logs 等）
```

---

## 2. 后端目录（Fastify）

```txt
apps/host-api/
  src/
    routes/          # 路由层
    controllers/     # 控制器层
    services/        # 业务服务层（用例与编排）
    providers/       # 技术能力提供者（基础设施与组合根接线）
    repositories/    # 存储访问层
    middlewares/     # 中间件
    schemas/         # 请求参数与配置 schema
    domain/          # 领域对象
    utils/           # 工具函数
  tests/
```

### 2.1 `providers` 与 `services` 的区别（定义与规范）

| 维度 | `providers/` | `services/` |
|------|----------------|---------------|
| **本质** | **能力提供者**：常驻对象、技术通道、事件总线、与框架无关的“底座” | **业务服务层**：用例、流程编排、领域规则与对外契约的落地 |
| **典型内容** | Host Event Hub、Notification 常驻投递、Hub 与各通道的 Bridge/Sink | 插件 Chat、AI Chat、调度器、租约、插件目录、LLM 运行时等 |
| **是否承载业务规则** | 原则上**不承载**业务用例（只做转发、订阅、过滤、去重等技术语义） | **承载**业务规则与跨仓储/跨插件的编排 |
| **谁创建、谁持有** | 在 **`app.ts`（组合根）** 中 `new` 并挂到 Fastify 实例或闭包，供路由注入 | 多为**无状态函数模块**或**由组合根注入依赖**的服务，不自行 new 全局 Hub |
| **依赖方向** | **禁止**依赖 `services/`、`routes/`、`controllers/`（避免环与业务泄漏） | **禁止**依赖 `fastify`、`routes/`、`controllers/`；可依赖 `repositories/`、可经组合根注入的发布函数 |
| **与 SQLite** | 不直接访问数据库 | 经 `repositories/` 访问 SQLite |

**明确定义**

- **`providers`（提供者）**：为应用提供**可复用的技术能力**与**进程级单例**（如事件总线、长连接通知的订阅与派发）。命名上强调“对外提供一种机制”，而不是“完成一次业务请求”。
- **`services`（业务服务）**：实现**一条业务用例或一条跨模块流程**（如“发送插件 Chat 并落库”“启动调度并执行插件任务”）。命名上强调“完成宿主对外承诺的行为”。

**后期规范（建议写入 PR 自检）**

1. 新增目录时先问一句：这是**通道/底座**还是**用例**？前者进 `providers/`，后者进 `services/`。
2. `providers` 内只做**接线（wiring）**与**技术策略**（过滤、背压可放服务或单独 `infra`，视复杂度而定）；业务分支（如“某类会话如何解析”）放在 `services`。
3. `services` 需要发布到 Hub/通知流时，**不直接 `new HostEventHub`**，由 `app.ts` 注入 `publish` 或薄封装（与当前 `registerSchedulerNotificationPublisher` 同类做法一致）。
4. `services` 若仅需要 `providers` 下的 **Type**，允许 `import type`；避免 `import` 具体类导致服务层反向依赖实现细节。
5. **`lint:arch` 已约束**：`services` **禁止**对 `providers` 做**值导入**（仅允许 `import type`）；`PluginRuntimeProvider` 等须在 **`app.ts`** 组合根构造，经 **`register*Routes(app, …)`** 传入路由闭包，再由路由**显式传参**调用 `services`（调度等无 HTTP 边界的启动逻辑由 **`app.ts` 直接向 service 传参**）。
6. **规则 A（`providers/*/index.ts`）**：每个 provider 子目录须有 **`index.ts`**；其中**运行时导出恰好 1 个**（`export class` 或 `export { SomeClass } from`，且 `export { … }` 内非 `type` 的标识符合计为 1）；**禁止** `export function` / `export const` / `export enum` / `export *` / `export default`；**允许**任意 `export type` 与 `export { type … } from`。由 `scripts/check-architecture.mjs` 扫描。

首批接口建议：
- 插件：`/api/plugins/*`
- LLM：`/api/llm/*`
- MCP：`/api/mcp/*`
- 插件会话：`/api/plugins/:pluginId/sessions/*`（通用，不做插件特判）

## 3. 前端目录（React + Vite + shadcn）

```txt
apps/host-console/
  src/
    pages/           # 页面级路由
    features/        # 业务模块（plugins/chat/weixin/llm）
    components/      # 通用组件 + shadcn ui
    lib/api/         # 接口调用封装
    app/             # router/provider
```

页面最小集合：
- `/plugins`：插件 Grid
- `/plugins/:pluginId/chat`：插件 Chat（插件驱动会话列表）
- `/settings/llm`：LLM custom 配置

---

## 4. 公共包目录

```txt
packages/
  protocol-core/     # 类型、错误码、契约
  plugin-sdk/        # 插件开发 SDK
  host-kernel/       # 宿主内核（调度/策略/会话）
  mcp-gateway/       # MCP 探测与调用
```

约束：
- 插件只能依赖 `plugin-sdk` 和 `protocol-core`。
- 插件不得依赖 `apps/host-api` 内部代码。
- 宿主前端不得出现 `weixin-*` 等插件特定组件目录，插件特性通过通用协议下发。

---

## 5. 插件目录

```txt
plugins/
  <plugin-id>/
    plugin.json
    src/
    dist/
```

示例：
- `plugins/weixin-bridge`
- `plugins/browser-command`
- `plugins/workspace-echo`

---

## 6. 初始化顺序

1. 先建 `apps/host-api` 与 `apps/host-console` 空壳并跑通。
2. 再建 `packages/protocol-core`，统一类型。
3. 再建 `packages/plugin-sdk`，固化插件接入方式。
4. 最后拆分 `host-kernel` 与 `mcp-gateway`。
